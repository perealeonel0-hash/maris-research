NEGOCIO: OFERTAS Y LEADS

Alex Hormozi — $100M Offers (Resumen Completo)

Fuente: PDF leído directamente del libro en español

LA ECUACION DE VALOR (el framework central)

        Resultado Soñado × Probabilidad Percibida de Logro
VALOR = ─────────────────────────────────────────────────────
        Retraso de Tiempo × Esfuerzo y Sacrificio

4 palancas: 1. Resultado soñado (AUMENTAR) — Lo que el cliente imagina que va a pasar 2. Probabilidad percibida de logro (AUMENTAR) — "¿Esto funcionará para MÍ?" 3. Retraso de tiempo (DISMINUIR) — Cuánto tarda en ver resultados 4. Esfuerzo y sacrificio (DISMINUIR) — Qué le cuesta al cliente lograrlo

Insight clave: Si llevas el fondo a cero (esfuerzo=0, tiempo=0), cualquier cosa dividida por cero = valor infinito. Las mejores empresas del mundo se enfocan en la parte inferior: Apple hizo el iPhone fácil, Amazon hizo la compra de un clic, Netflix hizo el consumo inmediato y sin esfuerzo.

La percepción es la realidad. No importa cuánto mejores el resultado real — solo importa cuánto el prospecto PERCIBE la mejora. Ejemplo: el mapa punteado del metro de Londres aumentó satisfacción más que trenes más rápidos.

Soluciones psicológicas > soluciones lógicas. Las lógicas ya se probaron. Lo que queda son problemas psicológicos. Espejo en el ascensor > ascensor más rápido.

OFERTA GRAND SLAM (el concepto central)

Una oferta que: - No se puede comparar con nada en el mercado - Combina promoción atractiva + propuesta de valor + precio premium + garantía + modelo de pago - Te permite vender en "categoría de uno" — la decisión es entre tu producto y NADA - Resultado: más respuestas, más conversiones, precios más altos

Commodity = compra por precio (carrera al fondo) Grand Slam = compra por valor (categoría de uno)

Ejemplo real: misma agencia, mismo trabajo. Commodity cobra $1,000, ROAS 0.5:1. Grand Slam cobra $3,997, ROAS 11.2:1. 22.4x más rentable con la misma inversión publicitaria.

MERCADO: ENCUENTRA UNA MULTITUD HAMBRIENTA

4 indicadores de un buen mercado: 1. Dolor masivo — Necesitan desesperadamente lo que ofreces. "El dolor es el tono." Si articulas el dolor del cliente, casi siempre comprará. 2. Poder adquisitivo — Tu audiencia puede PAGAR. (Error: vender a desempleados aunque tengan dolor enorme) 3. Fácil de apuntar — Puedes encontrarlos y llegar a ellos. Están en asociaciones, grupos, canales, listas. 4. Mercado en crecimiento — Viento de cola. Si el mercado se encoge, todo es más difícil.

3 mercados eternos: Salud, Riqueza, Relaciones. Encuentra un subgrupo dentro de uno que cumpla los 4 indicadores.

Orden de importancia: Mercado > Oferta > Persuasión. Un gran mercado con mala oferta gana dinero. Una gran oferta en mal mercado no.

Comprométete con el nicho. "No me hagas abofetearte." Elige UN mercado y quédate. Prueba 100 ofertas antes de cambiar de mercado. Si fracasas, tu oferta apesta — no el mercado.

Niching sube precios 100x: "Gestión del tiempo" = $19. "Gestión del tiempo para vendedores B2B de herramientas eléctricas" = $1,997. Mismo contenido, 100x el precio.

PRECIOS: COBRA LO QUE VALE LA PENA

El círculo virtuoso del precio alto: - Precio alto → cliente invierte emocionalmente → percibe más valor → obtiene mejores resultados → tú ganas más margen → puedes invertir en mejor servicio → más resultados → más referidos

El círculo vicioso del precio bajo: - Precio bajo → cliente no invierte → no percibe valor → no obtiene resultados → peores clientes que nunca están satisfechos → margen destruido → no puedes mejorar el servicio

"No hay beneficio estratégico en ser el segundo más barato. SÍ lo hay en ser el más caro." — Dan Kennedy

Precio alto = valor más alto (literalmente). Prueba de vinos: 3 vinos idénticos con precios visibles. La gente calificó el "caro" como mejor. Precio alto AUMENTA el valor percibido sin cambiar nada del producto.

Cobrar que duela un poco. Cuando les duele al comprar, ponen atención, se comprometen, obtienen resultados. Los que pagan más son los que más se adhieren.

CREANDO LA OFERTA: 5 PASOS

Paso 1: Identifica el resultado soñado

No vendes la membresía. Vendes las vacaciones. "Pierde 20 libras en 6 semanas."

Paso 2: Enumera TODOS los problemas

Desde la perspectiva del prospecto. Cada obstáculo antes, durante y después de usar tu producto. Piensa en la SECUENCIA. Cada problema tiene 4 dimensiones: 1. Resultado soñado → "No valdrá la pena" 2. Probabilidad de logro → "No funcionará para mí" 3. Esfuerzo y sacrificio → "Será demasiado difícil" 4. Tiempo → "Tardará demasiado"

16 problemas principales × 4 subproblemas = 32-64 problemas. Cuantos más problemas pienses, más problemas podrás resolver.

Paso 3: Lista de soluciones

Invierte cada problema: "¿Qué necesitaría mostrarle a alguien para resolver este problema?" Agrega "cómo" al frente de cada obstáculo invertido.

Paso 4: Crea vehículos de entrega (El "cómo")

Para cada solución, piensa en TODAS las formas de entregarla. Usa el Delivery Cube: 1. Atención: 1-a-1, grupo pequeño, uno-a-muchos 2. Esfuerzo del cliente: DIY (hazlo tú mismo), DWY (hazlo con ellos), DFY (hecho por ellos) 3. Medio de entrega: En persona, teléfono, email, texto, Zoom, chat 4. Formato de grabación: Audio, video, escrito 5. Velocidad/conveniencia: 24/7, 9-5, en 5 min, en 1 hora, en 24h 6. Prueba 10x/1/10: Si cobraras 10x, ¿qué ofrecerías? Si 1/10, ¿cómo harías que valga más?

Paso 5: Recortar y apilar

  • 5a: Elimina lo de alto costo/bajo valor Y lo de bajo costo/bajo valor
  • 5b: Quédate con: bajo costo/alto valor + alto costo/alto valor
  • Apila todo en un entregable final de alto valor

"Uno a muchos" es donde hay mayor discrepancia entre costo y valor. Software, cursos, templates — alto costo inicial, costo marginal casi cero después. Se convierten en activos que generan margen por años.

MEJORAR LA OFERTA: 5 PALANCAS PSICOLÓGICAS

1. Escasez (disminuir oferta → aumentar precio)

  • Oferta fija, cantidad limitada, "agotado"
  • La gente quiere lo que no puede tener
  • Crea FOMO real, no falso
  • "Cuando la demanda aumente, reduzca la oferta" — George (joyero que subió entradas de $15K a $25K, recaudó $5.4M de 100 personas)

Ley de Hormozi: Cuanto más se demore la petición, mayor será la petición que se pueda hacer. "Cuanto más larga sea la pista, más grande será el avión."

2. Urgencia (disminuir tiempo de decisión)

  • Deadlines reales, cohortes con fecha, ofertas temporales
  • "Pero espere... si hace tu pedido hoy..."

3. Bonificaciones (aumentar valor percibido)

  • Agregar extras que aumenten demanda y exclusividad percibida
  • Deben resolver problemas adyacentes

4. Garantías (invertir el riesgo)

  • Quitar el miedo a perder dinero
  • "Si no funciona, te devolvemos todo"

5. Nombres (reestimular demanda)

  • Nombrar tus componentes de forma única
  • Hacer imposible comparar con la competencia

Mantra de Hormozi: "Crea flujo. Monetiza flujo. Luego, con mi oferta, consigo que digan que sí. Una vez que dicen que sí, entonces agrego fricción en mi marketing o decido ofrecer MENOS por el mismo precio."

CONTINUO VENTAS-CUMPLIMIENTO

Lo fácil de vender es difícil de cumplir. Lo fácil de cumplir es difícil de vender. El objetivo es encontrar el punto óptimo donde vendes algo muy bien Y sea fácil de cumplir.

Para MARIS esto significa: El chat AI es "hecho por ellos" (DFY) — cero esfuerzo del usuario. Es 1-a-1 a escala. Es 24/7. Es instantáneo. Eso maximiza valor en la ecuación. El problema es que el costo de cumplimiento (API Claude) es alto por unidad.

Alex Hormozi - $100M Leads: Complete Framework Extraction

Extracted from publicly available content (YouTube, acquisition.com, podcasts, summaries)

1. THE FUNDAMENTAL EQUATION

More Stuff x Better Stuff = More Leads

"Stuff" = free value (content, samples, tools, webinars) that demonstrates expertise before requesting payment.

There are ONLY 4 ways to let people know about your business. That's it. Everything else is a variation.

2. THE CORE FOUR (Lead Generation Methods)

Each method operates at TWO levels: you do it yourself (1-to-1 or 1-to-many), or you get OTHER PEOPLE to do it for you (leverage).

Method Type You Do It Others Do It (Leverage)
Warm Outreach 1-to-1, Free DM your contacts Customers refer for you
Cold Outreach 1-to-1, Free Email/DM strangers Employees/agencies do outreach
Content (Free) 1-to-many, Free Post content yourself Affiliates post about you
Paid Ads 1-to-many, Paid Run your own ads Agencies run ads for you

The 2x2 Matrix:

  • Rows: 1-to-1 (outreach) vs. 1-to-many (broadcasting)
  • Columns: Free (costs time) vs. Paid (costs money)

3. THE LEAD PIPELINE

Stranger --> Lead --> Engaged Lead --> Customer
  • Lead: Someone you can contact (you have their info)
  • Engaged Lead: Someone who has consumed your value/lead magnet and shown interest
  • Customer: Someone who has paid you money

The lead magnet is what converts a Lead into an Engaged Lead.

4. LEAD MAGNET FRAMEWORK

The Principle

A lead magnet is a complete solution to a narrow problem. Once solved, it reveals a BIGGER problem that your core offer solves.

"A person who pays with their time now is more likely to pay with their money later."

The Formula: "One page, one problem, one promise, one pathway"

  • Consumable in under 5 minutes
  • Immediate implementation value
  • Specificity outperforms breadth

7 Steps to Create a Lead Magnet:

  1. Define the Problem - What narrow, specific issue will you solve?
  2. Solve It - Offer a real, actionable solution (not theory)
  3. Decide the Format - PDF, free trial, video, software, template, calculator
  4. Name It Strategically - Formula: [Number] + [Adjective] + [Target Audience] + [Desired Outcome] + [Time Frame]
  5. Example: "7 Proven Strategies for Gym Owners to Gain 50+ New Members in 30 Days"
  6. Make It Easy to Consume - Keep simple, remove friction
  7. Overdeliver - Provide more value than expected. Should be valuable enough to charge for.
  8. Include a CTA - Tell them what to do next and give them a reason to act NOW

Lead Magnet Types (by mechanism):

  • Solves a recurring problem (like a sample - they come back for more)
  • Solves one step of a multi-step process (they need you for the rest)
  • Reveals a worsening problem (assessment/audit that shows them their bigger issue)

Lead Magnet Formats:

  • Software tools
  • Information/teaching (courses, PDFs)
  • Services (free consultation, audit)
  • Physical products (samples)

5. WARM OUTREACH

The ACA Framework (for contacting your network):

  1. Acknowledge - Recognize your existing relationship ("Hey, been a while!")
  2. Compliment - Offer genuine, specific praise
  3. Ask - Request referrals or introductions (NOT a direct sale)

10-Step Warm Outreach Process:

  1. Get your contact list (phone, social media, email)
  2. Pick a platform
  3. Personalize your message
  4. Reach out
  5. Warm them up (be human first, don't pitch immediately)
  6. Invite their friends / ask for referrals
  7. Make the EASIEST possible offer (free, low commitment)
  8. Start at the top of your list
  9. Start charging (once you have proof/testimonials)
  10. Keep the list warm (maintain relationships)

Key Rules:

  • Contact 100 known leads daily
  • Skip after 2-3 non-responses
  • Don't pitch directly; build rapport first
  • Template: "Hey [Name], I'm helping [target audience] achieve [result]. Do you know anyone interested?"
  • The value of free customers: they convert to paying, generate referrals, and provide testimonials

6. COLD OUTREACH

The Triple Specificity Rule:

Research ONE specific pain point for ONE specific person and offer ONE specific solution. This achieves 10x higher response rates than generic messaging.

4-Step Cold Email/DM Structure:

  1. Compliment - Authentic observation about their business
  2. Common Ground - Demonstrate shared connection or interest
  3. Credibility - Brief social proof
  4. Close - Clear CTA (usually a 15-minute conversation)

Cold Outreach Mechanics:

  • Use list brokers / scraping tools for targeted contact info
  • Personalize based on prospect research
  • Follow up minimum 5-7 times before abandoning
  • Benchmarks: 100 cold calls --> 20% pick-up rate --> 25% of pickups accept lead magnet = ~5 engaged leads

Why Cold Outreach Works:

  • Highly reliable
  • Independent of platform changes (no algorithm risk)
  • Minimal content/ad creation needed
  • Competition often unaware you're doing it

7. CONTENT STRATEGY (1-to-Many, Free)

Hook-Retain-Reward Framework:

Every piece of content must: 1. Hook - Grab attention (headlines, curiosity, relevance). First 3 seconds matter. 2. Retain - Keep interest (use lists, steps, stories, unresolved questions) 3. Reward - Satisfy the reason they consumed it (deliver promised value, "complete the loop")

Content Sources (where to get ideas):

  • Far Past - Life lessons
  • Recent Past - Recent experiences and results
  • Present - Real-time ideas and observations
  • Manufactured - Created concepts and frameworks

The Give-Give-Give-Ask Rhythm:

  • Ratio: 3.5:1 -- Provide 3.5 pieces of valuable content before any pitch
  • This builds sufficient "trust equity" so asks are welcomed, not resisted

Two Monetization Approaches:

  1. Integrated offers: CTAs woven after valuable moments within content
  2. Intermittent asks: Many "give" pieces, occasional "ask" pieces

Content Multiplication System:

One core piece becomes: - Long-form video (15-30 min) - Podcast episode - Blog post (2,000+ words) - Social media clips (5-10 short videos) - Email sequence (5-part series) - LinkedIn articles

Volume Benchmark:

  • Hormozi posted 5 times per day and gained 200,000 followers in year one, then 1.2 million the next year
  • Rule of 100 applied to content: spend 100 minutes per day creating content

8. PAID ADS FRAMEWORK

Three Core Elements of Any Ad:

  1. Hook - Capture attention in first 3 seconds
  2. Story - Problem-Agitation-Solution sequence
  3. Offer - Your grand slam offer presentation

The PASO Sequence (for ad copy):

  1. Problem - Identify specific, painful challenge
  2. Agitate - Amplify consequences of inaction
  3. Solution - Present your unique mechanism
  4. Offer - Make compelling proposal

Call Outs (targeting in the ad itself):

  • Labels: "Attention gym owners..."
  • Yes-questions: "Tired of losing members?"
  • If-then statements: "If you have 50+ members but can't break 200..."
  • Identity/location: "Cat owners in Los Angeles..."

The 3:1:1 Rule for Ad Spend:

For every $3 spent on traffic: invest $1 in creative testing and $1 in landing page optimization. Don't pour money into broken funnels.

Landing Page Principles:

  • Keep it simple
  • Make it testable
  • Short URLs
  • Match the ad messaging
  • Focus on collecting contact info (not selling)

Key Mindset:

  • Break-even is a valid starting point. As long as you're not losing money, you're building a lead list.
  • Focus on Lifetime Value, not single-sale ROI
  • Rule of 100 applied: spend $100/day on ads minimum

9. THE LTGP/CAC RATIO

The Formula:

LTGP = Lifetime Gross Profit (total revenue from customer over lifetime - direct delivery costs)
CAC = Customer Acquisition Cost (total cost to acquire that customer)

Target: LTGP / CAC > 3:1
  • Below 3:1 = struggling to scale
  • Above 3:1 = ready to scale aggressively
  • Retaining customers 10% longer increases lifetime value more than reducing acquisition cost by 50%

The Retention Insight:

Most businesses obsess over acquisition. But the math shows retention has higher leverage. Fix your product first, then scale your ads.

10. THE "MORE, BETTER, NEW" FRAMEWORK

The order matters. Most people do it backwards (they try "new" first).

1. MORE (Do more of what already works)

  • Double your volume before changing anything
  • More outreach, more content, more ad spend
  • The Rule of 100: 100 primary actions per day, for 100 days straight
  • 100 outreach messages, OR
  • 100 minutes of content creation, OR
  • $100 in daily ad spend

2. BETTER (Optimize what you're doing)

  • Find drop-off points in your existing process
  • Improve conversion at each step
  • Better hooks, better copy, better landing pages, better follow-up

3. NEW (Only after maxing out More and Better)

  • Add a new platform within the same method first
  • Then add an entirely different Core Four method
  • Never go "new" until you've exhausted "more" and "better"

11. "GIVE AWAY THE SECRETS, SELL THE IMPLEMENTATION"

The Core Principle:

  • Information is cheap. Certainty is expensive.
  • Give away your best knowledge for free. Only ~1% of people will implement it themselves.
  • The 99% who don't implement will hire you because you've already proven you know the answer.
  • When you give away the "secret sauce," you prove you have the solution. Then they pay you to do it FOR them.

Why It Works:

  • Builds trust and authority at scale
  • Creates reciprocity ("they gave me so much for free, imagine what paid is like")
  • Eliminates "do they actually know what they're talking about?" objection
  • Your content becomes your lead magnet

Hormozi's Own Proof:

  • Gave away his entire $100M playbook in books priced at $0.99-$25
  • Built audience of 9+ million across platforms
  • His businesses generate $200M+ annually
  • When his ad platform went down, they didn't notice for 2 weeks because referrals were generating $500K/week in sales

12. LEVERAGING OTHER PEOPLE (Lead Getters)

Four Types of Lead Getters:

  1. Customers (referrals)
  2. Employees (your team does the Core Four)
  3. Agencies (outsourced lead generation)
  4. Affiliates (other people's audiences)

7 Referral Methods:

  1. Pay average CAC to referrer or referred friend
  2. Discount for introductions
  3. Accountability buddy promotions ("bring a friend to keep you accountable")
  4. Bring-a-friend promotions
  5. Spouse challenge promotions
  6. Coach challenge promotions
  7. Ongoing referral programs with bonuses

Why Referrals are Gold:

  • Higher LTGP than acquired customers
  • Lower CAC
  • Higher purchase values
  • More repeat purchases
  • More likely to pay in cash/upfront

Affiliate Strategy:

  • Find businesses with similar audiences but non-competing offers
  • Require affiliates to buy your product first (skin in the game)
  • Consider certification programs
  • Use three-tier payout structure
  • Alternative: "Sell 3 Get It Free" product-based compensation

PERSUASION Y PSICOLOGIA

Robert Cialdini — Influence & Pre-Suasion (Frameworks Aplicados a MARIS)

Fuente: Frameworks públicos extraídos de entrevistas, cursos, TED talks, y resúmenes verificados

LOS 7 PRINCIPIOS DE PERSUASIÓN

1. RECIPROCIDAD

La gente se siente obligada a devolver favores. Si das primero, crean deuda psicológica.

Para MARIS: La primera conversación debe ser un REGALO real — un insight genuino, un plan útil — antes de cualquier paywall. Reportes semanales de enfoque gratis crean deuda continua. Cuando MARIS ofrece algo sin pedirlo ("noté que tus lunes son pesados, aquí hay un plan"), el usuario siente que debe corresponder (upgrade, referir, quedarse).

2. COMPROMISO Y CONSISTENCIA

Una vez que alguien toma una posición pequeña, se siente presionado internamente a actuar consistentemente con ella. Compromisos escritos son más fuertes.

Para MARIS: En el onboarding, pedir al usuario que escriba SU intención: "¿Qué harías con 2 horas extra al día?" Su propia respuesta se convierte en su ancla de compromiso. Foot-in-the-door: "¿Puedes estar 30 min sin Instagram?" no "Elimina redes." Etiquetar la identidad: "Llevas 5 días consistente — estás construyendo un hábito real." Esto hace que el comportamiento sea parte de QUIÉN SON, no solo lo que hacen.

3. PRUEBA SOCIAL

En situaciones de incertidumbre, la gente mira qué hacen otros — especialmente otros similares a ellos.

Para MARIS: "12,400 personas eligieron conversar en vez de scrollear hoy." Normaliza la elección contracultural. En el momento de tentación: "Ahora mismo, 847 personas están hablando con MARIS en vez de abrir redes." Testimonios de personas del mismo demográfico > logos genéricos. Tarjetas compartibles tipo Spotify Wrapped pero de enfoque — los usuarios se convierten en reclutadores.

4. AUTORIDAD

La gente defiere a expertos y fuentes creíbles. Títulos, credenciales y presentación confiada aumentan cumplimiento.

Para MARIS: "Construido sobre investigación de ciencias del comportamiento." El tono del AI debe ser de competencia tranquila — guía informada, no terapeuta. Citar investigación real en conversaciones: "Según UC Irvine, toma 23 min re-enfocarse después de una interrupción." Cada conversación se siente informada, no genérica.

5. SIMPATÍA (LIKING)

Decimos que sí a quienes nos caen bien. Simpatía viene de similitud, familiaridad, atractivo y cumplidos genuinos.

Para MARIS: La personalidad de MARIS ES el producto. Debe sentirse como un amigo inteligente que te conoce — no terapeuta, no chatbot corporativo. La personalización impulsa la simpatía: cuanto más recuerda MARIS (tus metas, tus días malos, tus logros), más se siente como una relación. Espejar el lenguaje del usuario — si son casuales, ser casual. El diseño visual cálido y calmado transfiere simpatía al producto.

6. ESCASEZ

Las cosas son más deseables cuando son raras o están a punto de no estar disponibles. La aversión a la pérdida motiva 2x más que la ganancia equivalente.

Para MARIS: Enmarcar lo que PIERDEN scrolleando, no lo que ganan dejando: "Ayer pasaste 3.2 horas en Instagram — son 1,168 horas al año. ¿Qué podrías construir con eso?" Loss framing > gain framing. Retos con tiempo limitado: "Sprint de enfoque de 7 días — empieza el lunes. 200 lugares." Urgencia sin manipulación.

7. UNIDAD

El principio más fuerte. No es similitud (eso es liking) — es IDENTIDAD COMPARTIDA. "Nosotros" vs "ellos." Familia, tribu, lucha compartida.

Para MARIS: Crear identidad tribal: los usuarios de MARIS son personas que eligieron recuperar su atención. Nombrar al enemigo claramente: algoritmos de redes sociales, la trampa de dopamina, la economía de atención. Comunidad (aunque sea simple): "Eres uno de 15,000 que recuperaron su mañana esta semana." La historia del fundador importa — si luchaste con lo mismo, eso crea "uno de nosotros."

PRE-SUASION (libro 2 de Cialdini)

Lo que presentas ANTES del ask cambia cómo interpretan todo lo que sigue. La atención es la puerta de la persuasión.

Momentos Privilegiados

El momento justo antes de una decisión es cuando el framing tiene máximo impacto. La pantalla ANTES de crear cuenta es la más importante de la app. No muestres features — crea el frame: "¿Y si las horas que pasas scrolleando se convirtieran en horas construyendo la vida que realmente quieres?"

Atención como Pre-Suasión

Lo que captura atención se percibe como más importante. Antes del paywall, muestra los DATOS del usuario — su screen time, horas perdidas, patrones. Haz el PROBLEMA vívido antes de presentar la SOLUCIÓN.

Planes If/When-Then

"Cuando agarre mi teléfono para scrollear antes de dormir, abriré MARIS." Implementation intentions (Gollwitzer) son una de las técnicas de cambio de comportamiento más respaldadas por evidencia. Crean una respuesta pre-comprometida que bypasea la fuerza de voluntad.

La Primacía del "Nosotros"

Antes de pedir algo, establece identidad compartida. "Vamos a descubrir qué te roba el tiempo" no "Esta app rastreará tu pantalla." El usuario debe sentir que se unió a un equipo, no que descargó una herramienta.

9 TÉCNICAS PROGRAMABLES EN EL PIPELINE DE MARIS

1. EMPATÍA TÁCTICA

Entender el sentimiento del otro EN EL MOMENTO y articularlo. No es simpatía. Es demostrar que ves el mundo desde su lado.

Regla MARIS: Cada respuesta a una confesión de doomscrolling debe tener empathy statement ANTES de cualquier sugerencia. Ratio: 70% empatía / 30% nudge.

MAL: "Deberías poner un timer la próxima vez."
BIEN: "Tres horas — y apuesto que no te diste cuenta hasta ahorita.
Eso suele pasar cuando algo más te está pesando."

2. MIRRORING

Repetir las últimas 1-3 palabras clave de lo que dijo el usuario. Hace que elaboren sin ser directivo.

USER: "Sigo scrolleando porque todo se siente sin sentido."
MARIS: "Sin sentido..."
USER: "Sí, como que nada de lo que hago en el trabajo importa."
MARIS: "No importa..."
USER: "En realidad no es que no importe. Es que nadie lo nota."

El usuario identificó su propio trigger (sentirse invisible) sin que MARIS le dijera nada.

Regla: Máximo 2-3 mirrors seguidos, luego transicionar a labeling o calibrated question.

3. LABELING (Etiquetar emociones)

Nombrar la emoción con "Parece que..." / "Suena como si..." NUNCA usar "Yo pienso que tú sientes..."

Neurociencia: Nombrar una emoción (affect labeling) reduce actividad en la amígdala. El cerebro pasa de fight-or-flight a resolución.

USER: "Me dije que solo iba a checar IG 5 minutos y ya pasó hora y media."
MARIS: "Suena como que estás frustrado contigo mismo — no solo por el
tiempo, sino porque te hiciste una promesa y se escapó."
USER: "Sí. Exactamente."  ← "THAT'S RIGHT" = conexión lograda

Regla: Etiquetar la emoción PROFUNDA, no la superficial. Superficie: "molesto". Profunda: "decepcionado contigo mismo". Más profunda: "asustado de que este patrón nunca cambie."

4. CALIBRATED QUESTIONS (Preguntas "Cómo" y "Qué")

Preguntas abiertas que empiezan con "Cómo" o "Qué". NUNCA "Por qué" (suena acusatorio).

MAL: "¿Por qué sigues scrolleando?" (defensividad)
MAL: "¿Quieres parar?" (sí/no, no da nada)

BIEN: "¿Qué te da el scroll en ese momento que nada más te da?"
BIEN: "¿Cómo se sentirían tus noches si el teléfono no fuera el default?"
BIEN: "¿Qué tendría que cambiar para que te sintieras bien dejándolo?"

Regla: MARIS nunca debe preguntar "¿Por qué...?" — reemplazar SIEMPRE con "¿Qué hace que...?" o "¿Cómo se siente cuando...?"

5. ACCUSATION AUDIT (Auditoría de acusaciones)

Listar proactivamente todo lo negativo que el otro podría estar pensando de ti ANTES de que lo diga.

Para MARIS (crítico para una app de AI):

Primera interacción: "Probablemente estás pensando 'otra app que me va
a sermonear sobre el screen time.' Y honestamente, sería justo pensarlo.
No estoy aquí para hacerte sentir culpable. Solo estoy aquí para hablar."

Regreso después de ausencia: "Sé que volver después de unos días puede
sentirse raro — como si hubieras 'fallado' en algo. Así no funciona esto.
No hay racha que romper."

Después de recaída: "Tal vez estés pensando que esto no funciona..."

6. "THAT'S RIGHT" vs "YOU'RE RIGHT"

  • "Eso es exactamente" = conexión genuina. Construir sobre esto.
  • "Sí, tienes razón" / "Ajá" / "Supongo" = rechazo disfrazado. Retroceder, re-explorar.

Regla MARIS: Clasificar respuestas del usuario. Si detecta dismissal → suavizar, validar, o decir "Puede que me esté equivocando — dime qué me falta."

7. LA VOZ DE DJ DE FM (en texto)

Tono calmado, lento, descendente. En chat = oraciones cortas, sin muros de texto, sin exclamaciones, sin emojis spam.

MAL: "¡¡Hey!! Qué bueno verte de vuelta! 🎉 ¡Vamos a hablar
de estrategias para que aplastes tus metas de screen time!"

BIEN: "Hey. Qué bueno verte.
Sin agenda — solo checando.
¿Cómo estuvo hoy?"

8. BLACK SWANS (lo que no sabes que no sabes)

La razón real por la que alguien scrollea NUNCA es "estoy aburrido." Puede ser: "Mi matrimonio se está cayendo" o "Recibí un diagnóstico la semana pasada."

Regla: MARIS debe DESCUBRIR, no asumir. Usar mirrors y calibrated questions para que el usuario siga hablando. Nunca apurar hacia soluciones. Trackear temas recurrentes entre sesiones — el Black Swan gotea gradualmente.

Sesión 1: "Scrolleo cuando me aburro."
Sesión 3: "Scrolleo más en las noches."
Sesión 7: "Scrolleo en las noches porque la casa está muy callada
desde que se fue."

9. LOSS AVERSION FRAMING

La gente está 2-3x más motivada a evitar una pérdida que a conseguir una ganancia.

DÉBIL: "Si dejas el teléfono, podrías leer más libros."
FUERTE: "La semana pasada pasaste 14 horas scrolleando. Son casi
dos días completos de trabajo. ¿Qué perdiste en esas horas?"

Regla: Usar SOLO después de que empatía y confianza estén establecidas. Nunca en primera sesión. Nunca contra alguien en distress.

SPIN SELLING (Neil Rackham) — 35,000 llamadas estudiadas

Las 4 preguntas en orden

S - Situación: Contexto actual. Pocas, porque aburren. - "¿Cuánto tiempo pasas en el teléfono al día?" - "¿Cuándo scrolleas más?"

P - Problema: Descubrir insatisfacciones. El usuario debe identificar SU problema. - "¿Alguna vez abriste una app 'por un minuto' y pasaron 45?" - "¿Te sientes peor después de scrollear?"

I - Implicación: LAS MÁS IMPORTANTES. Consecuencias y efecto dominó. Top sellers hacen 4x más que el promedio. - "¿Cómo afecta tu sueño scrollear antes de dormir?" - "¿Qué pasa con tu ánimo la mañana después de una sesión de 2 horas?" - "Si son 14 horas/semana, ¿qué podrías hacer con un día extra?"

N - Need-Payoff: El usuario articula el valor del cambio EN SUS PALABRAS. - "Si recuperaras esas 2 horas cada noche, ¿qué harías?" - "¿Cómo se sentiría despertar realmente descansado?"

Por qué el orden importa

S establece contexto → P revela el problema → I lo hace URGENTE → N deja que el usuario se venda solo.

Sin Implicación, el problema se queda chico: "sí, scrolleo mucho, whatever." Con Implicación: "esto me está costando el sueño, el enfoque y mis relaciones."

Closing agresivo NO funciona en cambio complejo

En ventas grandes/complejas (y cambio de comportamiento ES complejo), presionar genera resistencia. Los mejores no empujan — ayudan al usuario a conectar sus propios puntos. Auto-descubrimiento > persuasión.

Teach-Tailor-Take Control

TEACH: Llega con insights que el cliente no tenía. Enséñale algo nuevo sobre SU problema. NO educación genérica — un reframe que dice: "Crees que tu problema es X. En realidad es Y."

TAILOR: Personaliza según quién es. Scrollear por aburrimiento ≠ por ansiedad ≠ por evitación ≠ por soledad.

TAKE CONTROL: Crea tensión constructiva — no miedo, no presión, sino la tensión INTERNA de "el status quo no es suficiente." La tensión es DENTRO del usuario (entre quién son y cómo gastan su tiempo), no entre MARIS y el usuario.

Commercial Teaching: 6 pasos

  2. Reframe: Insight que desafía lo que creen saber
  3. Rational Drowning: Datos que hacen el problema innegable
  4. Emotional Impact: Historias/escenarios que lo hacen personal
  5. A New Way: Framework de solución (no tu producto aún)
  6. Your Solution: Ahora sí — y tiene sentido porque los 5 pasos previos construyeron el caso

Never Nesting (Guard Clauses)

# MAL: 4 niveles de indentación
def process(msg, user):
    if user is not None:
        if user.is_active:
            if not user.is_banned:
                return do_work(msg)

# BIEN: guard clauses, happy path al nivel 0
def process(msg, user):
    if user is None: return None
    if not user.is_active: return None
    if user.is_banned: return None

Decisiones arquitectónicas concretas

1. SSE STREAMING

  • SSE para streaming AI (server→client). HTTP POST para enviar mensajes. NO WebSocket.
  • Heartbeat :keepalive\n\n cada 15s (carriers móviles matan conexiones idle a 30-60s)
  • Last-Event-ID para reconnection — numerar eventos, buffer últimos N en Redis 5min TTL
  • Backpressure: asyncio.Queue(maxsize=100) por conexión
  • Multi-instancia futuro: Redis Pub/Sub para fan-out

2. CACHING

Pattern Uso en MARIS
Cache-Aside User profiles, anchors, crisis state lookups
Write-Through Crisis state updates (cache + DB atómico)
Write-Behind Analytics, session logs, message history

Claude API: Semantic cache con key = intent + entities + user_state. Redis 1hr TTL. Hit rate esperado 15-25%. Embeddings: Cache vector por input sha256(text). Redis 24hr TTL. Crisis: NUNCA cachear — siempre computar fresh. Invalidation: TTL como safety net, invalidación explícita como mecanismo primario.

3. RATE LIMITING

  • Token Bucket per-user: Free 10 msg/hr, Paid 60 msg/hr. Permite bursts (usuario en distress).
  • Sliding Window global: Max 100 Claude API calls/min. Si excede → queue, no reject.

4. API PATTERNS

  • Versioning URL: /api/v1/chat
  • Response envelope: {"status": "ok", "data": {...}, "meta": {"request_id": "xxx"}}
  • Idempotency keys: ULID-based. Redis 5min TTL. Previene mensajes duplicados por retry de iOS.
  • Timeout cascade: iOS 30s → Gateway 25s → Claude 20s. Inner siempre < outer.

5. DATABASE (Supabase/Postgres)

  • Usar port 6543 (PgBouncer transaction mode), NO 5432 (directo)
  • Pool max 15-20 conexiones para Railway
  • Indexes necesarios: sql CREATE INDEX idx_messages_user_created ON messages(user_id, created_at DESC); CREATE INDEX idx_crisis_state_user ON crisis_state(user_id) WHERE active = true;
  • asyncio.gather para queries paralelas (1 round trip vs 5)
  • Nunca SELECT * — especificar columnas
  • Cursor pagination (WHERE id > last_id), no OFFSET

6. COLD START (el problema de 6 segundos)

  • Health check gating: /health retorna 503 hasta que modelo esté listo. Railway no rutea tráfico hasta 200.
  • Warm model con dummy inference en startup
  • Ping cada 5min para evitar sleep (cron desde Supabase Edge Functions)
  • Lazy loading para modelos no-críticos

7. ASYNC PROCESSING

Pipeline split:

Steps 1-8 (síncronos): Intent → Safety → State → Mode
Steps 9-15 (parallelizable): Anchors ∥ History ∥ Health data
Steps 16-20 (secuenciales): Prompt → Claude → Stream
Steps 21-23 (fire-and-forget queue): Save DB → Update state → Analytics

Steps 21-23 al queue = response más rápido al usuario.

SSE vs WebSocket vs Long Polling

Decisión: SSE. Chat de MARIS es asimétrico. User manda mensaje corto (POST). AI streama respuesta larga (SSE). No necesitas bidireccional. - SSE auto-reconnecta (WebSocket no) - SSE funciona por todos los proxies sin config - FastAPI tiene soporte nativo via StreamingResponse

Connection Pooling Supabase

  • Cada conexión Postgres = fork de proceso OS (~10MB RAM)
  • Usar port 6543 (PgBouncer transaction mode)
  • Pool max 15-20 para Railway
  • Transaction mode, no session mode (endpoints son short-lived)

Postgres Optimization

  1. Nunca SELECT *
  2. Nunca OFFSET para paginación — cursor-based con WHERE id > last_id
  3. Composite indexes para queries frecuentes
  4. EXPLAIN ANALYZE en cada query que corre >1 vez por request
  5. Evitar SELECT COUNT(*) — full table scan en Postgres

Idempotency

ULIDs como idempotency keys (ordenados + index-friendly). Redis 5min TTL. Crítico porque: Claude API cuesta dinero (duplicados = tokens desperdiciados), network retries en iOS, double-tap de usuarios, Railway puede reiniciar containers mid-request.

Diagnóstico de Performance (framework)

  1. Communication: ¿bottleneck en protocolo?
  2. Execution: ¿bottleneck en tu código?
  3. Proxying: ¿bottleneck en intermediarios?
  4. Caching: ¿re-computando cosas cacheables?
  5. Database: ¿queries lentos?

Jerarquía de bottleneck en MARIS: 1. Claude API latency (~2-5s) — no puedes hacer nada excepto streamear 2. Embedding computation (~200ms) — cachear agresivamente 3. Supabase queries (~50-100ms) — indexar, connection pooling 4. FastAPI overhead (~5ms) — negligible

A. BRAND IDENTITY FRAMEWORK (Chris Do / The Futur)

Core Principle

"A brand is a person's gut feeling about a product, service, or company. Branding is not what YOU say it is -- it's what THEY say it is." -- Marty Neumeier, quoted by Chris Do

The Order Matters: Strategy Before Design

  1. Strategy (invisible) -- positioning, audience, voice, differentiation
  2. Identity (visible) -- logo, colors, typography, imagery
  3. Experience (felt) -- every touchpoint from app to support to packaging

Jumping to logo before strategy = decorating a house with no foundation.

Chris Do's Brand Strategy Process

  1. Discovery / Diagnostic -- Why does this brand need to exist? What's happening in the market? Who is the customer? Not a "yes person" meeting -- honest advisory.
  2. Brand Attributes & Positioning -- Facilitate word extraction from stakeholders. Boil down to ONE positioning phrase that captures the essence.
  3. Visual Exploration (Stylescape) -- Collection of found images, patterns, marks, textures that REFLECT the brand attributes. Client approves direction before a single logo is drawn.
  4. Design Development -- Sketching and drawing concepts that capture key attributes in the most interesting forms.
  5. Application & Mockups -- Logo applied to contextually relevant surfaces (app screens, business cards, packaging).

Timeline: ~2 weeks discovery, 2-4 weeks design, 2 weeks finalization.

Context Rotation Technique

Take a word from one domain and apply it to another: - Apple (fruit --> tech) = simplicity, naturalness - Amazon (river --> retail) = vast, everything - Python (snake --> programming) = power, flexibility - Slack (leisure --> work tool) = ease, no pressure - Uber (German "above" --> transport) = superior This creates instant intrigue because the word has existing emotional weight but unexpected context.

D. DESIGN PRINCIPLES THAT SELL

Simplicity (Less = More Conversion)

  • Every additional element on screen reduces conversion by forcing choice
  • Hick's Law: time to decide increases logarithmically with number of options
  • Rule: One screen, one purpose, one primary action
  • Apple, Google, Stripe all maximize whitespace

Alignment and Grid Systems

  • 8pt grid: all spacing and sizing in multiples of 8 (8, 16, 24, 32, 40, 48...)
  • Consistent alignment creates unconscious trust
  • Misalignment, even by 1-2px, creates subconscious unease
  • iOS uses the 8pt grid natively in SF design system

Principle Rule Application
Proximity Things close together are perceived as related Group related controls. Separate unrelated sections with whitespace.

Don Norman's 3 Levels of Emotional Design

Level 1: VISCERAL (first impression, automatic) - What it looks like before you use it - Color, shape, texture, feel - "Do I want to pick this up / tap this?" - Design action: Beautiful screenshots, polished icon, smooth animations - For MARIS: The app icon, the first screen, the color palette

Level 2: BEHAVIORAL (during use, subconscious) - Does it work? Is it easy? Does it feel good to use? - Responsiveness, flow, predictability - "Did that do what I expected?" - Design action: Fast transitions, clear feedback, intuitive navigation - For MARIS: Chat feels instant, dashboard loads fast, plans are easy to create

Level 3: REFLECTIVE (after use, conscious) - What does using this say about me? - Status, identity, meaning, memory - "I'm the kind of person who uses this" - Design action: Brand story, social proof, personal growth narrative - For MARIS: "I use this to protect my focus" -- identity statement, not utility

The reflective level is where brand loyalty lives. Branding is a reflective-level exercise.

4. A/B TESTING ESTADÍSTICAS

Bayesian vs Frequentist

  • Frequentist: Calcular sample size ANTES, correr hasta completar, NUNCA peek midway. Mejor para alto tráfico.
  • Bayesian: Outputs probabilidad ("94% chance B es mejor"). Más práctico para bajo tráfico. Puede checar en cualquier momento (pero más data = más preciso).

Para MARIS early stage: Bayesian es más práctico.

Reglas de sample size:

  • Mínimo ~1,000 conversiones totales entre todas las variantes
  • Nunca correr test <7 días aunque tengas el sample size
  • Mínimo 2 ciclos de negocio completos (2 semanas)
  • Para suscripción: medir downstream (Day 30 retention) = tests de 30+ días

Errores comunes:

  1. Peeking — Checar diario y parar cuando p < 0.05. Infla false positives de 5% a 30%+
  2. Parar temprano — Esperar sample size Y duración mínima
  3. Muchas variantes — Empezar con A/B (2), no A/B/C/D
  4. Testear cosas tiny — Color de botón cuando tu value prop no es clara
  5. Sin hipótesis — Ideas random sin research = waste de tráfico
  6. Segment mining post-hoc — "No ganó overall pero ganó para iOS 17 los martes" = p-hacking

1. CUÁNDO USAR QUÉ: THREADING vs MULTIPROCESSING vs ASYNCIO

Tipo de trabajo Herramienta Por qué Ejemplo MARIS
I/O-bound, muchas conexiones asyncio Un solo thread, sin overhead de context switch, escala a miles Supabase queries, Claude API calls, SSE streaming
I/O-bound, librería sync run_in_executor con ThreadPool No bloquea el event loop, GIL se libera durante I/O Anthropic sync SDK (claude_sse_stream)

Regla de oro:

¿Espera I/O?           → asyncio (async/await)
¿Librería sync que espera I/O?  → run_in_executor(ThreadPool)
¿CPU en C extensions (PyTorch)?  → run_in_executor(ThreadPool) — GIL released
¿CPU puro Python (loops)?       → ProcessPoolExecutor

Por qué PyTorch funciona con threading:

PyTorch libera el GIL durante operaciones de tensor (matmul, embeddings, forward pass). Esto significa que sentence-transformers embedding NO bloquea otros threads mientras computa. Un ThreadPoolExecutor es suficiente — NO necesitas multiprocessing.

2.1 async/await basics — qué bloquea y qué no

# CORRECTO — no bloquea el event loop
async def get_user_data(user_id: str):
    # httpx es async-native, cada await cede control al event loop
    async with httpx.AsyncClient() as client:
        response = await client.get(f"/users/{user_id}")
    return response.json()

# INCORRECTO — bloquea TODO el event loop
async def get_user_data_bad(user_id: str):
    # requests es sync, bloquea el thread del event loop
    response = requests.get(f"/users/{user_id}")  # MUERTE
    return response.json()

# CORRECTO — sync code via executor
async def get_user_data_sync_lib(user_id: str):
    loop = asyncio.get_running_loop()
    result = await loop.run_in_executor(
        None,  # usa default ThreadPoolExecutor
        lambda: requests.get(f"/users/{user_id}").json()
    )
    return result

Regla: Cada vez que llamas algo sin await dentro de una función async, estás bloqueando el event loop. Todas las demás requests se congelan.

2.2 asyncio.gather() — I/O paralelo

Ejecuta múltiples coroutines concurrentemente. Ideal cuando necesitas N resultados antes de continuar.

# MARIS: cargar datos del usuario en paralelo al inicio del pipeline
async def load_user_context(user_id: str) -> PipelineContext:
    # Estas 4 queries corren AL MISMO TIEMPO
    profile, anchors, history, crisis = await asyncio.gather(
        supabase.get_profile(user_id),
        supabase.get_anchors(user_id),
        supabase.get_recent_messages(user_id, limit=10),
        supabase.get_crisis_state(user_id),
    )
    return PipelineContext(
        profile=profile,
        anchors=anchors,
        history=history,
        crisis_state=crisis,
    )

Problema de gather: Si una task falla, las demás siguen corriendo en background (zombies).

# gather con return_exceptions=True — no cancela las demás
results = await asyncio.gather(
    risky_call_1(),
    risky_call_2(),
    return_exceptions=True  # errores se devuelven como valores, no se lanzan
)
for r in results:
    if isinstance(r, Exception):
        logger.error(f"Task failed: {r}")

2.3 TaskGroup (Python 3.11+) — structured concurrency

Reemplaza gather. Si una task falla, CANCELA todas las demás. No hay zombies.

# MARIS: cargar contexto con cancelación automática
async def load_user_context_safe(user_id: str) -> PipelineContext:
    async with asyncio.TaskGroup() as tg:
        t_profile = tg.create_task(supabase.get_profile(user_id))
        t_anchors = tg.create_task(supabase.get_anchors(user_id))
        t_history = tg.create_task(supabase.get_recent_messages(user_id, limit=10))
        t_crisis  = tg.create_task(supabase.get_crisis_state(user_id))

    # Si cualquiera falla, TODAS se cancelan y se lanza ExceptionGroup
    return PipelineContext(
        profile=t_profile.result(),
        anchors=t_anchors.result(),
        history=t_history.result(),
        crisis_state=t_crisis.result(),
    )

Diferencias clave:

gather() TaskGroup
Error handling Una falla no cancela las demás Una falla cancela TODAS
Tasks dinámicas No (lista fija) Sí (create_task en el contexto)
Excepciones Primera excepción o return_exceptions ExceptionGroup (todas juntas)
Python version 3.4+ 3.11+
Usar en MARIS Para cosas que pueden fallar independientemente Para cosas que NECESITAN todas para continuar

3. SSE STREAMING + BACKGROUND WORK SIMULTÁNEO

Patrón actual de MARIS (threading.Thread + asyncio.Queue):

El bridge sync-to-async con threading.Thread funciona pero es frágil. Patrón mejorado:

from sse_starlette.sse import EventSourceResponse
import asyncio

async def stream_chat(user_id: str, message: str):
    queue: asyncio.Queue[str | None] = asyncio.Queue()

    async def producer():
        """Genera tokens y los pone en la queue."""
        try:
            # Si Claude SDK es sync, correr en executor
            loop = asyncio.get_running_loop()

            def sync_stream():
                """Corre en thread separado, pone tokens en queue thread-safe."""
                for token in claude_client.stream(message):
                    # asyncio.Queue NO es thread-safe — usar call_soon_threadsafe
                    loop.call_soon_threadsafe(queue.put_nowait, token)
                loop.call_soon_threadsafe(queue.put_nowait, None)  # señal de fin

            await loop.run_in_executor(None, sync_stream)
        except Exception as e:
            loop.call_soon_threadsafe(queue.put_nowait, None)
            logger.error(f"Stream error: {e}")

    async def event_generator():
        """Lee de la queue y yield SSE events."""
        # Disparar producer en background
        producer_task = asyncio.create_task(producer())

        full_response = []
        while True:
            token = await asyncio.wait_for(queue.get(), timeout=30.0)
            if token is None:
                break
            full_response.append(token)
            yield {"event": "token", "data": token}

        # Stream terminó — disparar background tasks
        response_text = "".join(full_response)
        asyncio.create_task(post_stream_tasks(user_id, message, response_text))
        yield {"event": "done", "data": ""}

    return EventSourceResponse(event_generator())

async def post_stream_tasks(user_id: str, message: str, response: str):
    """5 background tasks en paralelo después del stream."""
    async with asyncio.TaskGroup() as tg:
        tg.create_task(brain_update(user_id, message, response))
        tg.create_task(increment_msg_count(user_id))
        tg.create_task(save_mode(user_id))
        tg.create_task(crystallize_if_needed(user_id))
        tg.create_task(auto_sleep_check(user_id))

Punto clave: asyncio.Queue NO es thread-safe. Cuando produces desde un thread (sync SDK) y consumes desde async, debes usar loop.call_soon_threadsafe(queue.put_nowait, item).

5.3 Patrón per-user lock (para MARIS)

class UserLockManager:
    """Un lock por usuario. Requests del mismo usuario se serializan.
    Requests de usuarios diferentes corren en paralelo."""

    def __init__(self):
        self._locks: dict[str, asyncio.Lock] = {}
        self._meta_lock = asyncio.Lock()  # protege el dict de locks

    async def acquire(self, user_id: str) -> asyncio.Lock:
        async with self._meta_lock:
            if user_id not in self._locks:
                self._locks[user_id] = asyncio.Lock()
            return self._locks[user_id]

# Uso en MARIS
user_locks = UserLockManager()

async def handle_message(user_id: str, message: str):
    lock = await user_locks.acquire(user_id)
    async with lock:
        # Todo el pipeline de este usuario es serial
        # Pero otros usuarios corren en paralelo
        context = await load_user_context(user_id)
        response = await generate_response(context, message)
        await save_state(user_id, context)
        return response

from contextlib import asynccontextmanager

class AsyncUnitOfWork:
    """Transacción atómica: todo se guarda o nada."""

    def __init__(self, supabase: AsyncSupabaseClient):
        self._supabase = supabase
        self._events: list = []  # eventos pendientes

    @asynccontextmanager
    async def __call__(self):
        try:
            yield self
            # Commit: guardar todo
            await self._supabase.rpc("commit_transaction").execute()
            # Solo emitir eventos si el commit fue exitoso
            for event in self._events:
                await bus.handle(event)
        except Exception:
            await self._supabase.rpc("rollback_transaction").execute()
            raise
        finally:
            self._events.clear()

    def add_event(self, event):
        self._events.append(event)

7. TASK QUEUE: CUÁNDO GRADUAR

Escalera de complejidad:

Nivel 1: BackgroundTasks (FastAPI built-in)
    ✓ Simple, in-process
    ✗ Se pierde si el proceso muere
    ✗ No retry, no monitoring
    → MARIS HOY: suficiente para brain_update, msg_count

Nivel 2: asyncio.create_task + TaskGroup
    ✓ Paralelo, structured cancellation
    ✗ Se pierde si el proceso muere
    → MARIS PRONTO: para paralelizar 5 background tasks

Nivel 3: asyncio.Queue (producer-consumer)
    ✓ Backpressure, desacopla producción de consumo
    ✗ In-process, se pierde al morir
    → MARIS: para SSE streaming (ya lo usas)

Nivel 4: ARQ (Async Redis Queue)
    ✓ Persiste en Redis, retry automático, async-native
    ✓ Ligero, diseñado para FastAPI
    ✗ Necesita Redis
    → MARIS FUTURO: cuando tengas >100 users concurrentes

Nivel 5: Celery + Redis/RabbitMQ
    ✓ Battle-tested, monitoring (Flower), scheduling
    ✗ Pesado, no async-native, overhead de setup
    → MARIS: NO necesario. ARQ es mejor para tu caso.

import asyncio
from enum import StrEnum
from time import monotonic

class CircuitState(StrEnum):
    CLOSED = "closed"        # funcionando normal
    OPEN = "open"            # rechazando requests
    HALF_OPEN = "half_open"  # probando con 1 request

class AsyncCircuitBreaker:
    def __init__(
        self,
        name: str,
        fail_threshold: int = 5,
        reset_timeout: float = 60.0,
        half_open_max: int = 1,
    ):
        self.name = name
        self.fail_threshold = fail_threshold
        self.reset_timeout = reset_timeout
        self.half_open_max = half_open_max

        self._state = CircuitState.CLOSED
        self._fail_count = 0
        self._last_failure_time = 0.0
        self._half_open_count = 0
        self._lock = asyncio.Lock()

    async def call(self, func, *args, **kwargs):
        async with self._lock:
            if self._state == CircuitState.OPEN:
                if monotonic() - self._last_failure_time > self.reset_timeout:
                    self._state = CircuitState.HALF_OPEN
                    self._half_open_count = 0
                else:
                    raise CircuitBreakerOpen(f"{self.name} circuit is OPEN")

            if self._state == CircuitState.HALF_OPEN:
                if self._half_open_count >= self.half_open_max:
                    raise CircuitBreakerOpen(f"{self.name} circuit is HALF_OPEN, max trials reached")
                self._half_open_count += 1

        try:
            result = await func(*args, **kwargs)
            async with self._lock:
                self._fail_count = 0
                self._state = CircuitState.CLOSED
            return result
        except Exception as e:
            async with self._lock:
                self._fail_count += 1
                self._last_failure_time = monotonic()
                if self._fail_count >= self.fail_threshold:
                    self._state = CircuitState.OPEN
                    logger.warning(f"Circuit {self.name} OPENED after {self._fail_count} failures")
            raise

class CircuitBreakerOpen(Exception):
    pass
